### 背景知识：web3j-crypto 和 web3j-core 不一样

Plan 1 已引入 `web3j-crypto`，它只负责 BIP32/签名/哈希等密码学能力，不能发 RPC。Plan 2 需要 `web3j-core` 调 `eth_getBlockByNumber`、`eth_call`、`eth_sendRawTransaction`、`eth_feeHistory` 等 JSON-RPC。

### 设计原理与决策

- `wallet.chain.eth.enabled=false` 时不装配 ETH Bean，避免没有 anvil/geth 时启动失败。
- 配置用 `@ConfigurationProperties`，不把 RPC URL 写死在代码里。
- 本期先实现一个 endpoint 的稳定路径；多 RPC 路由类先落接口和权重模型，真正熔断/限流在后续 Task 逐步补齐。

## Phase 2 — ETH ChainClient：链上读与交易状态查询

### 新增功能

实现 `ChainClient`：查最新高度、查区块、查余额、查链上 pending nonce、查交易状态。

### 背景知识：EVM 节点读接口

- `eth_blockNumber`：当前最新高度。
- `eth_getBlockByNumber(height, true)`：拿块和交易列表。
- `eth_getTransactionReceipt(txHash)`：判断交易是否被打包、是否成功。
- `eth_getTransactionCount(address, pending)`：拿包含 mempool pending 交易后的下一个 nonce，适合启动校准。
- `eth_getBalance(address, latest)`：查 ETH 原生币余额。
- ERC20 余额不是账户字段，要调合约 `balanceOf(address)`。

### 设计原理与决策

- `queryTxStatus` 只把链状态映射为 Plan 1 的 `TxStatus`，不修改业务表。
- `getBlock` 返回 `ChainBlock`，raw JSON 保留下来供 scanner 审计回放。
- `getBalance(address, coinSymbol)` 对 `ETH` 查原生余额；ERC20 余额由合约配置驱动，具体 `contract` 由后续 `CoinEntity` 查询补齐。

- 为什么 receipt.status=0 的交易必须跳过
- `latest` nonce 与 `pending` nonce 的区别
- 为什么 scanner 需要保留 raw JSON

## Phase 3 — ETH 地址派生、签名与 EIP-1559 构造

### 新增功能

实现 `AddressDerivator`、`ChainSpecificSigner`、`TxBuilder`、`TxBroadcaster`，让业务能构造、签名、广播 ETH 原生币和 ERC20 转账。

### 背景知识：EIP-1559 交易字段

EIP-1559 交易不再只填 `gasPrice`，而是：

- `maxFeePerGas`：你愿意支付的最高单价。
- `maxPriorityFeePerGas`：给 validator 的小费。
- `baseFee`：协议按区块拥堵自动计算并销毁。
- 实际支付约等于 `min(maxFeePerGas, baseFee + priorityFee) * gasUsed`。

ERC20 转账不是 ETH value 转账，而是调用合约：

```
transfer(address to, uint256 amount)
methodId = a9059cbb
```

交易的 `to` 是 ERC20 合约地址，真正收款地址在 calldata 里。

### 设计原理与决策

- `EthTxBuilder` 不访问私钥，只构造 `RawTx`。
- `EthChainSpecificSigner` 接收短生命周期 privateKey byte[]，签完返回 `SignedTx`，不记录明文。
- `RawTx.chainSpecific` 放 `maxFeePerGas/maxPriorityFeePerGas/gasLimit/chainId/contract/calldata`，业务层不解析。
- 广播失败分两类：节点暂时失败抛 `EthRpcException`；nonce too low / insufficient funds 这类业务失败交给 withdraw 状态机落失败原因。

### 面试考点

- ETH 原生转账 vs ERC20 转账的数据结构差异
- EIP-1559 为什么要 `baseFee * 2 + priorityFee`
- 为什么 signer 不放在 withdraw 里
- 为什么预测 txHash 可以在广播前算出

### 新增类清单

| 类 | 职责 |
|---|---|
| `EthAddressDerivator` | secp256k1 公钥 -> Keccak -> EIP-55 地址 |
| `EthTxBuilder` | 构造 ETH/ERC20 EIP-1559 RawTx |
| `EthChainSpecificSigner` | web3j 签名 EIP-1559 raw transaction |
| `EthTxBroadcaster` | `eth_sendRawTransaction` 广播 |

### Task 清单

- [ ] **Task 45**: `EthTxBuilder` 支持 ETH 原生转账 + 单测。

## Phase 4 — EthFeeStrategy：EIP-1559 手续费估算

### 新增功能

业务层调用 `FeeStrategyRegistry.quote(req)` 即可拿到 ETH/ERC20 fee quote。

### 背景知识：手续费估算为什么不能写死

ETH baseFee 会按区块拥堵动态变化。写死 gasPrice 的系统在拥堵时会长时间卡 pending，在空闲时又多付手续费。EIP-1559 的常见工程策略是 `maxFeePerGas = baseFee * 2 + priorityFee`，在未来几个块 baseFee 连续上涨时仍有缓冲。

### 设计原理与决策

- 用 `eth_feeHistory(20, latest, [25,50,75])` 取最近 20 块费用历史。
- priorityFee 取 50 分位，避免极端高价污染。
- `gasLimit`：ETH 原生 21000；ERC20 默认 65000，后续可用 `eth_estimateGas` 精确化。
- `max_gas_price_gwei` 超上限时拒绝报价，提现建单失败，不冒险烧过高 gas。

### 面试考点

- EIP-1559 的 baseFee burn / priorityFee 激励
- fee 上限保护属于风控还是链策略
- fee 估算误差由谁兜底

### 新增类清单

| 类 | 职责 |
|---|---|
| `EthFeeStrategy` | ETH/ERC20 EIP-1559 报价 |
| `EthFeeQuoteKeys` | `chainSpecific` key 常量 |

### Task 清单

## Phase 5 — Scanner 框架与 ETH 解析

### 新增功能

实现链无关扫块框架、cursor 持久化、ETH 区块解析、ERC20 Transfer Log 解析、特殊交易过滤。

### 背景知识：scanner 不是简单定时器

扫块器要同时解决 4 件事：

1. **不漏**：cursor 持久化，重启后从上次高度继续。
2. **不重**：`chain_tx.uk(chain, tx_hash, vout)` 做幂等闸。
3. **不误判**：失败交易、self-transfer、合约创建、internal tx 都要有处理策略。
4. **抗 reorg**：只扫 `latest - reorgDepth` 的安全高度，并主动检查 parentHash。

### 设计原理与决策

- 新增 `scanner_cursor` 表比复用 `chain_tx` 更清晰：cursor 是扫描进度，不是交易事实。
- `AbstractScanner` 负责调度、cursor、幂等、发布事件；`EthTxParser` 负责链特化解析。
- ETH 原生转账从 block transaction 解析；ERC20 从 receipt logs 解析。
- `vout` 对 ETH 原生固定 0；ERC20 用 `logIndex`，支持同一 tx 多个 Transfer。

### 面试考点

- 为什么 scanner 要保存 cursor
- ERC20 Transfer Log 的 topic0/topic1/topic2/data 含义
- 为什么 receipt.status=0 必须过滤
- safeHeight 与主动 reorg 检测的区别

### 新增表

V3 迁移新增：

```sql
CREATE TABLE scanner_cursor (
  chain VARCHAR(16) PRIMARY KEY,
  last_scanned_height BIGINT NOT NULL,
  last_scanned_hash VARCHAR(128),
  updated_at DATETIME(3) NOT NULL
);
```

## Phase 6 — 充值确认与入账

### 新增功能

把 scanner 发现的 `chain_tx` 转成 `deposit_order`，确认数达标后调用 `LedgerService.credit` 入账。

### 背景知识：充值为什么不能扫到就入账

链上交易刚打包仍可能因为 reorg 失效。交易所通常等待 N 个确认后才给用户可用余额。ETH 常见确认数是 12。本项目用 `chain_config.deposit_confirms` 控制。

### 设计原理与决策

- `DepositDetectedHandler` 幂等创建 `deposit_order(PENDING)`。
- `DepositConfirmJob` 定时刷新 `confirm_count`，达到确认数后发布 `wallet.deposit.confirmed`。
- `DepositCreditHandler` 同事务更新订单为 `CREDITED` + `ledger.credit` + outbox 发布 `wallet.account.credited`。
- 找不到 `wallet_address` 的交易只存 `chain_tx`，不创建充值订单。

### 面试考点

- 充值确认数为什么按链配置
- 为什么入账必须通过 `LedgerService.credit`
- reorg 后已经 CREDITED 的订单如何 reverseCredit

|---|---|
| `deposit/DepositStatus` | PENDING/CONFIRMED/CREDITED/REVERSED/FAILED |
| `deposit/DepositDetectedHandler` | detected 事件 -> deposit_order |
| `deposit/DepositConfirmJob` | 刷确认数 |
| `deposit/DepositConfirmedEvent` | 确认事件 |

## Phase 7 — Reorg 主动检测与回滚

### 新增功能

scanner 扫块前主动检查 parentHash；发现分叉后回退 cursor、标记 orphaned、对已入账充值做反向冲账。

### 背景知识：reorg 对交易所有什么影响

如果用户充值所在块后来被孤立，账本里已经给用户加的钱就不再有链上依据。正确做法不是删除流水，而是追加一组反向 journal：这符合 append-only 审计原则。

### 设计原理与决策

- `ReorgHandler` 从当前高度往前找共同祖先。
- `(forkPoint, oldHead]` 的 `chain_tx` 标记 ORPHANED。
- `deposit_order.CREDITED` 走 `ledger.reverseCredit`，订单改 `REVERSED`。
- `withdraw_order.BROADCASTED/CONFIRMED` 回到 BROADCASTING，等待同 txHash 重新确认。

### 面试考点

- reorg 与 pending 的区别
- 已入账充值为什么不能 delete，而要 reverseCredit
- 超深 reorg 为什么要停机人工介入

### 新增类清单

| 类 | 职责 |
|---|---|
| `scanner/ReorgHandler` | 分叉检测与回滚 |
| `scanner/ReorgDetectedEvent` | 告警/审计事件 |

## Phase 8 — riskbridge 与 risk 钱包规则

### 新增功能

wallet 内定义风控防腐层，risk 模块实现钱包规则：单笔额度、日累计、频控、黑名单、大额人工复核。

### 背景知识：为什么 wallet 不直接写风控逻辑

钱包提现流程要稳定，风控规则会频繁变化。`wallet.riskbridge.RiskGuard` 是防腐层：wallet 只关心 `PASS/REJECT/MANUAL`，risk 模块可以以后接 AML、KYT、后台规则引擎，但不污染提现状态机。

### 设计原理与决策

- 本期 risk 模块使用内存配置 + Mapper 查询 withdraw_order 统计，不新增复杂规则表。
- 黑名单先用配置列表，后续可迁移 DB。
- `MANUAL` 不继续签名，订单进入 `MANUAL_REVIEW`；审批接口可留最小 service 方法，不做 admin 页面。

### 面试考点

- 防腐层为什么放 wallet 内而不是 risk 内
- REJECT 与 MANUAL 的状态差异
- 风控必须在冻结后还是冻结前：本期建单时先 freeze 再风险推进，拒绝时 unfreeze，保证订单资金被锁住

| `wallet/riskbridge/WithdrawCheckCtx` | 风控上下文 |
| `wallet/riskbridge/RiskDecision` | PASS/REJECT/MANUAL |
| `risk/wallet/DefaultWalletRiskGuard` | risk 模块默认实现 |

## Phase 9 — 提现状态机与事件驱动主路径

### 新增功能

实现提现建单、冻结、风险推进、签名、广播、确认、结算，所有异步推进通过 outbox + 幂等 handler。

### 背景知识：提现为什么要状态机

提现不是一个数据库事务能完成的动作。它跨越风控、签名、链节点广播、链上确认。状态机的价值是把"订单现在处于哪一步、下一步允许做什么、失败怎么恢复"表达清楚，并让超时巡检可以安全重放事件。

状态流：

```
INIT -> RISK_CHECKING -> SIGNING -> BROADCASTING -> BROADCASTED -> CONFIRMED -> SETTLED
       └ REJECTED      └ FAILED     └ FAILED        └ FAILED
       └ MANUAL_REVIEW -> SIGNING
```

### 设计原理与决策

- `WithdrawApplicationService.apply` 同事务：insert `withdraw_order(INIT)` + `ledger.freeze` + outbox `wallet.withdraw.init`。
- 每个 handler 都先 CAS 更新状态，CAS 失败视为幂等命中。
- `SignHandler` 分配 nonce、构造 rawTx、调用 Signer，回填 `signed_raw/predicted_tx_hash/nonce/from_address`。
- `BroadcastHandler` 广播后回填 `tx_hash`，进入 BROADCASTED。
- `WithdrawConfirmHandler` 由 scanner/confirm job 推进 CONFIRMED。
- `SettleHandler` 调 `ledger.settle`，订单 SETTLED。

### 面试考点

- 为什么不能同步阻塞到链上确认
- 状态机幂等三道闸：order CAS、tx_hash uk、journal uk
- nonce 分配失败是否回滚 freeze：签名前失败应 unfreeze 或进入 FAILED，由巡检补偿

## Phase 10 — 加速、取消与 nonce gap 处置

### 新增功能

实现 EVM 同 nonce 替换交易：加速提现、取消提现。

### 背景知识：EVM 的"撤销"其实是替换

EVM 账户同一 nonce 只能最终上链一笔交易。如果旧交易 pending 太久，可以发同 nonce、gas 更高的新交易替换它：

- **加速**：同 from/to/amount，同 nonce，提高 fee。
- **取消**：同 nonce，to=from，value=0，只消耗 gas，把 nonce 占掉。

链上没有真正"撤销交易"这个操作。

### 设计原理与决策

- 加速新建一条 `withdraw_order(replace_of_id=oldId)`，旧订单进入 REPLACED。
- 取消只允许 admin 工具触发，创建 CANCEL 类型订单，不给用户可见余额重复变动。
- 替换交易复用旧 nonce，不调用 `NonceAllocator.allocate`。

### 面试考点

- nonce gap 如何产生
- 加速和取消为什么都是同 nonce 替换
- 为什么取消交易 to=from value=0

## Phase 11 — anvil 端到端验收

### 新增功能

用本地 anvil 节点跑真实链路：ETH 充值、ERC20 充值、ETH 提现、ERC20 提现、reorg、加速。

### 背景知识：为什么用 anvil

anvil 是 Foundry 提供的本地 EVM 节点，启动快、可控、支持 `evm_mine`、`evm_snapshot`、`evm_revert`，非常适合钱包 e2e 和 reorg 注入。

### 验收标准

- `mvn -pl wallet -am test` 绿。
- anvil e2e：
  - 用户充值 ETH -> scanner detected -> deposit CREDITED -> account available 增加。
  - 用户充值 ERC20 -> scanner detected -> deposit CREDITED。
  - 用户提现 ETH -> freeze -> PASS -> sign -> broadcast -> confirmed -> settle。
  - 用户提现 ERC20 同样跑通。
  - pending tx 加速成功替换。
  - reorg 注入后已入账充值 reverseCredit。
- bootstrap 启动正常，`/actuator/health` UP。
